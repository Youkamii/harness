# claude-harness — 내 전용 자율형 Claude Code 하네스

## 하네스가 뭔가?

**하네스(harness)** 는 AI 모델을 감싸는 "장구(裝具)"다. 승마 하네스가 말의 힘을 마차로 전달하고 폭주를 막듯이, AI 하네스는 모델의 능력을 실제 작업으로 연결하고 사고를 막는다.

핵심 공식은 **`에이전트 = 모델 + 하네스`** — 모델(Claude)이 아닌 전부가 하네스다. 같은 모델이라도 어떤 규칙·안전장치·작업 절차를 둘러주느냐에 따라 결과물의 품질이 완전히 달라진다. 2026년 업계의 합의: *"좋은 모델 + 평범한 하네스보다, 평범한 모델 + 좋은 하네스가 낫다."*

이 하네스는 실행 프로그램이 아니라 **설정 파일 묶음**이다. Claude Code(엔진)에 얹는 규칙과 지식의 저장소이며, 엔진 실행 자체는 Claude Code 본체가 점점 잘하게 되므로 우리는 "무엇을 지킬 것인가"에만 투자한다.

## 세 가지 원칙

1. **자율** — 사용자에게 묻지 않고 스스로 판단해 진행한다. 확인은 딱 3가지(비용 발생·복구 불가 파괴·외부 공개)만. 단, 요청 범위를 넘겨짚어 확장하지 않는다.
2. **검증** — 실행해본 것만 "완료"다. 자기가 짠 코드에 자기가 관대해지는 문제를 막기 위해, 별도 AI(서브에이전트)들이 적대적으로 검증한다.
3. **집행** — 반드시 지킬 규칙은 프롬프트(부탁)가 아니라 훅(코드)으로 막는다. 부탁은 잊혀도 코드는 무조건 실행된다.

## 어떻게 작동하나 — 시점별 전체 그림

```
세션 시작 ──→ [헌법] CLAUDE.md가 Claude의 기본 규칙으로 주입됨
   │
Claude가 셸 명령 실행 시도 ──→ [훅] 명령이 실행되기 "직전"에 검사
   │                            ├─ 재앙급(루트 삭제, main 강제 푸시…) → 차단(deny)
   │                            ├─ 중위험(rm -rf, reset --hard…)     → Claude에게만 경고 주입(warn)
   │                            └─ 그 외                              → 통과
   │
기능 개발 요청 ──→ [스킬] /kickoff 파이프라인이 절차를 강제
   │               태스크 분해 → GitHub 이슈 → 검증 → 기능별 커밋 → 적대 리뷰 → 푸시·이슈 닫기
   │
코드 완성, 푸시 전 ──→ [서브에이전트] /red-review가 리뷰어 3명을 병렬 소환
   │                    (정합성·보안·단순화 렌즈) → 주장 재확인 → 진짜 결함만 수정
   │
작업 종료 ──→ [스킬] /wrapup — 검증 상태 정리, 남은 일 이관, 교훈을 메모리에 저장
```

## 구성 요소 상세

### 1. 헌법 — `CLAUDE.md`

**작동 방식**: 설치하면 `~/.claude/CLAUDE.md`에 마커 블록(`<!-- G-HARNESS BEGIN/END -->`)으로 병합되고, Claude Code가 **매 세션 시작 때 자동으로 읽어** 기본 행동 규칙이 된다. 별도 실행이 필요 없다.

담고 있는 규칙:
- **자율 원칙**: 묻지 말고 판단하되 넘겨짚지 말 것, 오토 퍼미션 존중(훅·스킬 어느 층위도 사용자를 호출하지 않음), 질문+작업이 같이 오면 질문 먼저.
- **사실/추측 구분** (최우선): 모든 판단을 **확인됨**(직접 실행해서 봄) / **추정**(근거 있는 추측) / **미확인** 셋 중 하나로 표시. "아마 될 겁니다"를 "됩니다"로 줄여 말하는 것을 금지.
- **파이프라인 순서**와 **안전 수칙**(비밀은 .env로 분리 등).

### 2. 훅 — `hooks/` (자동 안전장치)

**작동 방식**: 훅(hook)은 "특정 시점에 자동 실행되는 작은 프로그램"이다. Claude가 셸 명령을 실행하려는 **직전**에 Claude Code가 이 스크립트를 호출하고, 실행하려는 명령 내용을 넘겨준다. 스크립트가 "차단"이라고 답하면 그 명령은 아예 실행되지 않는다. Claude의 기억이나 선의에 의존하지 않는 **강제 장치**다.

| 훅 | 검사 시점 | 하는 일 |
|---|---|---|
| `guard.js` | 모든 셸 명령 직전 | **차단(deny)**: 루트/홈/시스템 폴더 통삭제(`rm -rf /`, `/*`, 대문자 `-Rf` 우회 포함), main/master 강제 푸시, 디스크 포맷, 포크밤, **새 터미널 창 스폰**(`Start-Process cmd/powershell/pwsh/wt`, `start` 별칭 — 작업 중 콘솔 창을 화면에 띄우지 않는다). **경고(warn)**: 일반 `rm -rf`, `git reset --hard`, `git clean -f`, 강제 푸시, 숨김 플래그 없는 `Start-Process` — 실행은 허용하되 Claude에게 "경로/창 여부를 다시 확인하라"는 메모를 주입 |
| `secrets-guard.mjs` | `git add`/`commit` 직전 | 커밋될 파일들에서 API 키(AWS·GitHub·OpenAI류…)·개인키·`.env` 파일을 스캔해 발견 시 **커밋 차단**. `commit -am`, `cd 다른폴더 && commit` 같은 우회 경로도 추적 |
| `format-changed.mjs` | `Edit`/`Write` 직후 | 프로젝트 루트에 `format-hook.config.json`이 있을 때만(opt-in) 방금 편집된 파일 하나를 프로젝트 포매터로 정리. 항상 비차단 |

중요한 설계 결정: **훅은 사용자를 절대 호출하지 않는다.** 예전엔 중위험 명령에서 사용자에게 확인창을 띄웠는데, 오토 퍼미션을 준 의미가 없어지므로 "차단 아니면 조용한 경고" 2단계로 바꿨다.

**품질 보증**: 훅마다 회귀 테스트가 있고(케이스 목록은 `hooks/*.test.*`가 정본), 설치기가 테스트 실패 시 배포를 거부한다. 패턴을 고칠 땐 테스트 케이스부터 추가하는 게 규칙.

### 3. 스킬 — `skills/` (작업 절차서)

**작동 방식**: 스킬은 "이런 상황에선 이 절차대로 하라"는 마크다운 문서다. `~/.claude/skills/<이름>/SKILL.md`로 배포되면 Claude Code가 자동 인식한다. 사용자가 `/이름`으로 직접 호출할 수도 있고, Claude가 상황 설명(description)을 보고 스스로 꺼내 쓸 수도 있다.

| 스킬 | 언제 | 무엇을 |
|---|---|---|
| `/kickoff` | 새 기능/프로젝트 구현 시작 | 계획 없이 코드부터 짜는 것을 금지하는 파이프라인: ① 요구사항 해석·완료기준(DoD) 정의 → ② 기능 단위 태스크 분해 → ③ GitHub 이슈 생성 → ④ 한 번에 한 기능씩 구현, **실행 검증 통과 후 기능별 커밋**(#이슈 연결) → ⑤ 전 기능 완료 후 적대 리뷰 → ⑥ 수정 커밋·푸시·이슈 닫기 |
| `/red-review` | 코드 완성 후, 푸시 전 | 아래 서브에이전트 항목 참고. 리뷰 비용 폭주 방지 규칙 내장(최대 3라운드, 푸시 전 1회가 기본) |
| `/wrapup` | 작업 마무리 | 변경 요약(커밋 연결), 항목별 확인됨/미검증 표시, 남은 일을 이슈 등 **영속 매체**로 이관, 재사용할 교훈을 메모리에 저장, 다음 세션 시작 문장 제안 |
| `/harness` | 하네스 점검/수정 | 훅 등록·테스트·실발화(진짜 차단되는지) 확인, 새 스킬/패턴 추가 절차, 레포↔배포본 동기화 |

### 4. 서브에이전트 — `agents/` (독립 검증자들)

**작동 방식**: 서브에이전트는 **별도 대화방에서 소환되는 독립 Claude**다. 본체가 작업한 맥락을 모르는 "신선한 눈"이라, 자기 코드에 관대해지는 문제가 구조적으로 차단된다. `~/.claude/agents/<이름>.md`로 배포되면 본체가 필요할 때 소환한다.

| 에이전트 | 역할 |
|---|---|
| `red-reviewer` | 적대적 리뷰어. 렌즈(정합성/보안/단순화) 하나를 지정받아 결함만 찾는다. 모든 지적은 "파일:줄 + 실패 시나리오" 형태로 반증 가능해야 하고, 확신 없으면 PLAUSIBLE로 구분 표시. 병렬 3개가 동시에 돌므로 **테스트 실행은 금지**(서로 간섭 방지) — 정적 분석만 하고, 재현은 본체가 순차로 한다 |
| `verifier` | 독립 검증자. "코드 읽어보니 될 것 같다" 판정이 금지된 에이전트 — 반드시 실제로 실행(테스트/빌드/구동)해서 DoD 항목별로 확인됨/실패/미검증을 판정한다 |

리뷰 흐름: 본체가 리뷰어 3개를 병렬 소환 → 보고를 모아 **직접 파일을 열어 재확인** → 재현되는 것(CONFIRMED)만 수정, 나머지는 기각 사유 명시. 리뷰어의 주장을 그대로 믿지 않는 것까지가 설계다.

### 5. 설치기 — `install.mjs`

**작동 방식**: 이 레포(소스)를 `~/.claude`(배포본)에 복사·병합한다. 7단계:

```
① 회귀 테스트 실행 (guard·secrets·contracts·smoke·constitution — 실패하면 중단)
② 잔존 자산 정리 — 소스에서 삭제된 harness 파일을 배포본에서도 제거
③ skills/ agents/ hooks/ contracts/ 를 ~/.claude/ 아래로 복사
④ 설치 스냅숏(assets-manifest.json)을 배포본에만 저장 (다음 설치의 잔존 검출 기준)
⑤ settings.json에 훅 설정 병합 (파일이 손상돼 있으면 덮어쓰지 않고 중단)
⑥ CLAUDE.md에 헌법 블록 병합 (마커 사이만 교체 — 마커가 비정상이면 중단)
⑦ 배포본 검증 — 실시간 해시로 복사가 실제 반영됐는지 대조
```

manifest는 저장소에 커밋하지 않는다 — 소스 이력은 git이 정본이고, 스냅숏은 배포본 쪽에만 필요하다 (적대 리뷰 판정).

**멱등(idempotent)** 하다 — "몇 번을 다시 실행해도 결과가 같고 안전하다"는 뜻. 기존에 쓰던 다른 스킬·설정은 건드리지 않는다.

```powershell
# Windows            # macOS/Linux
.\install.ps1        ./install.sh     (둘 다 Node.js 필요)

node install.mjs --verify   # 아무것도 바꾸지 않고 드리프트(소스↔배포본 불일치)만 검사
```

### 6. 계약 — `contracts/`

모든 저장 워크플로우가 지키는 입출력 계약(JSON Schema). 핵심: **CONFIRMED 판정은 외부 증거(evidence+verifiedBy) 없이 스키마를 통과할 수 없다** — "동일 모델 다수결은 검증이 아니다" 원칙의 집행 장치. 상세는 [`contracts/README.md`](contracts/README.md).

## 유지보수 규칙

- **이 레포가 소스, `~/.claude`는 배포본.** 항상 레포를 고치고 install로 배포한다. 배포본만 고치면 다음 설치 때 사라진다.
- 가드 패턴 수정: `hooks/*.test.*`에 케이스 먼저 추가 → 수정 → ALL PASS → 배포.
- 훅을 추가할 땐 "모델의 어떤 한계 때문에 존재하는가"를 파일 상단 주석으로 남긴다 — 모델이 좋아지면 그 훅의 제거를 검토할 수 있는 목록이 된다.

## 지원 범위 (capability)

이 하네스가 어디까지 동작하는지의 경계. 자동화를 설계할 때 이 표를 벗어난 가정을 하지 않는다.

| 영역 | 지원 | 비고 |
|---|---|---|
| OS | Windows 11 (주 사용), macOS/Linux (install.sh) | 훅은 node로 실행되어 OS 무관 |
| Claude Code 기능 의존 | 훅(PreToolUse), 서브에이전트(agents/*.md), 스킬(skills/*/SKILL.md), 저장 워크플로우(.claude/workflows) | 워크플로우는 Workflow 도구가 있는 버전 필요 |
| 클라우드 루틴(routines) | **로컬 `~/.claude` 접근 불가** — 선택한 저장소만 복제해서 실행 | 하네스·메모리 유지보수는 반드시 로컬(예약 작업·세션)에서 |
| 워크플로우 스크립트 | 파일시스템 접근 불가, Date.now/Math.random 불가 | 계약 스키마는 작성 시점에 인라인 임베드 |
| 상한(비용) 집행 | Workflow 도구 내장 budget·동시성 캡 사용 | 자체 상한 엔진을 만들지 않는다 |
| 드리프트 감지 | `node install.mjs --verify` (무토큰, 로컬) | 해시는 개행 정규화(CRLF→LF) 후 계산 |

## 알려진 한계 (의도된 것)

- guard는 명령 **문자열 전체**를 검사하므로, `echo`나 커밋 메시지 속에 위험 패턴이 글자로 들어 있어도 차단될 수 있다. 따옴표 안을 봐주기 시작하면 `bash -c "..."` 우회가 뚫리기 때문에 일부러 보수적으로 둔다. 오탐을 만나면 위험 문자열을 파일에 써서 우회하면 된다.
- warn 단계는 실행을 막지 않는다 — 사용자를 방해하지 않는 대가로, 최종 방어는 deny 목록과 Claude 자신의 판단에 맡긴다.
- **원천 차단 불가한 우회 3종(문자열 검사의 한계, 의도적 미차단)**: ① 난독화 실행 `echo <base64> | base64 -d | sh` — 디코드 전엔 평문에 위험 패턴이 없다. ② 따옴표 분할 `r""m -rf /` — 셸이 접합하지만 정규식은 못 본다. ③ 스크립트 래퍼 — 훅은 Bash/PowerShell 툴 명령만 보므로 래퍼 내부의 위험 명령·커밋은 못 본다. ①②는 guard/secrets가 아니라 Claude 자신의 판단과 헌법이 막는 영역이다. ③의 **커밋 경로에는 저장소별 방어선이 있다**: `node scripts/install-precommit.mjs <저장소>`로 git pre-commit 훅을 설치하면(옵트인) 래퍼 경유 커밋도 git 자신이 secrets-guard로 검사한다 — 스캔 로직은 secrets-guard.mjs 재사용이라 패턴 이중 관리가 없다.
- **적대 감사 이력(2026-07-11, Phase 2)**: 크로스벤더 red-review(Claude 서브에이전트 + Codex 재현)로 guard 우회 13종·secrets 갭 7종을 실제 명령으로 재현·확인한 뒤 전부 메웠다(테스트에 회귀 케이스로 고정). 재현 불가한 위 3종만 한계로 남겼다.
