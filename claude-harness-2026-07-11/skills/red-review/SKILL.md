# red-review — 멀티에이전트 적대 리뷰

자기가 짠 코드에 자기가 관대해지는 문제를 **컨텍스트 분리**로 막는다. 리뷰어는 구현 과정을 모르는 신선한 눈이다.

## 1. 대상 확정

- 리뷰 범위 diff를 정한다: 기본은 `git diff <기준브랜치>...HEAD` + 미커밋 변경. 기준이 없으면 최근 작업 커밋들.
- 대상 파일 목록과 변경 요약을 만든다 (리뷰어 프롬프트에 넣을 것).

## 2. 병렬 소환 — 서로 다른 렌즈 3개

Agent tool로 red-reviewer 서브에이전트 3개를 **한 메시지에서 동시에** 소환한다. 렌즈가 다르면 중복 리뷰가 아니라 상호 보완이다:

| 렌즈 | 집중 대상 |
|---|---|
| 정합성 | 로직 버그, 엣지케이스, 요구사항(DoD)과 실제 동작의 불일치 |
| 보안 | 입력 검증, 비밀 노출, 인젝션, 권한, 파일/네트워크 접근 |
| 단순화 | 죽은 코드, 중복, 과설계, 더 짧고 명확한 대안 |

각 프롬프트에는 반드시 포함: 대상 경로/diff 범위, 렌즈 지정, DoD, "반증 가능한 형태(파일:라인 + 실패 시나리오)로만 보고" 지시.

**리뷰어에게 테스트/빌드 실행을 지시하지 않는다** — 병렬 리뷰어들이 같은 작업 트리를 공유하므로 동시에 상태를 바꾸면(포트·캐시·산출물) 서로 간섭해 유령 실패를 결함으로 보고하게 된다. 리뷰어는 정적 분석까지만, 실행 재현은 3단계에서 메인이 직접 순차로 한다.

## 3. 판정 — 리뷰어를 믿지 않는다

- 발견 전부를 모아 중복 제거 후, **직접 해당 파일을 열어 재확인**한다.
- 판정: **CONFIRMED**(재현 시나리오가 성립) / **기각**(사유 명시). critical/major는 재확인 없이 채택 금지.
- 렌즈 2개 이상이 같은 지점을 지적하면 최우선으로 다룬다.

## 4. 수정과 보고

- CONFIRMED만 심각도순으로 수정하고, 수정 후 관련 테스트/실행 재검증.
- 보고: 발견 n건 → CONFIRMED m건(수정 완료) / 기각 k건(사유). **"문제없음"이라고만 말하지 않는다** — 발견이 없으면 무엇을 어떤 렌즈로 검사했는지 보고한다.

## 강도 조절

- 일반 기능: 렌즈 3개 1라운드.
- 중요 릴리스/보안 민감: 수정 후 라운드를 반복하되 **최대 3라운드**. 종료 조건은 "새 **CONFIRMED** 0건" — PLAUSIBLE 의심은 라운드를 연장하지 않는다 (적대 리뷰어는 언제나 새 의심을 만들어내므로, 의심 기준으로는 영원히 안 끝난다).
- 사소한 변경(오타, 문서): 리뷰 생략 가능 — 생략했음을 보고에 명시.
