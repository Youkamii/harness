# cross-review — 두 모델 교차 검증/리뷰

서로 다른 두 프론티어 모델(codex `gpt-5.5`/xhigh + `claude-opus-4-8`)이 같은 프로젝트를 **독립적으로** 검사한다. 한 모델의 맹점을 다른 모델이 잡는 구조 — 하네스의 "작업자/평가자 분리" 원칙을 모델 차원으로 확장한 것.

## 모드 선택

- **verify (검증)**: 코드를 짜기 전/중에 방향을 검사. 요구사항-문서-태스크 정합성, 설계 논리, "작성돼 있으나 나중에 문제 될 부분"을 적대적으로 의심.
- **review (리뷰)**: 개발 진행 중/후에 결과물을 검사. 문서·태스크 기준으로 커밋·이슈·코드를 읽고 진행 정합성, 코드 품질, 보안, 버그 가능성.
- 사용자가 명시하지 않으면 상황으로 판단: 코드가 거의 없으면 verify, 커밋이 쌓여 있으면 review, 애매하면 둘 다 순차 실행.

## 절차

1. **컨텍스트 수집.** 대상 프로젝트에서 다음을 모아 `/tmp/cross-review-context.md` 하나로 작성한다:
   - 요구사항·문서: README, docs/, 스펙 문서, `.harness/features.json`·progress.md (있는 것만)
   - 구조: 주요 디렉토리 트리 (3depth 이내)
   - 이력: `git log --oneline -30`, 열린 이슈 (`gh issue list` — gh 저장소일 때만)
   - 사용자가 특별히 지목한 관심사가 있으면 맨 위에 "중점 검사 요청" 으로 명시
   - 컨텍스트가 클 필요는 없다 — 리뷰어들도 파일을 직접 읽을 수 있으므로, 길잡이가 되는 핵심만 담는다
2. **백그라운드 실행.** 이 스킬 폴더의 `run-reviewers.mjs`를 백그라운드로 실행한다 (두 모델이 병렬로 돌며 수 분~수십 분 걸릴 수 있다):
   ```bash
   node <스킬폴더>/run-reviewers.mjs --mode verify --context /tmp/cross-review-context.md --cwd <대상폴더>
   ```
   실행 중에는 다른 작업을 계속해도 된다. 완료되면 `SYNTHESIS_READY` 와 보고서 경로 2개가 출력된다.
3. **종합.** 두 보고서(`.harness/reviews/<시각>-<모드>-{codex,claude}.md`)를 모두 읽고 하나로 합친다:
   - **양쪽 모두 발견** → 신뢰도 높음, 최우선 보고
   - **한쪽만 발견** → 직접 해당 파일/문서를 열어 사실 확인 후 채택/기각 (기각 사유 명시)
   - **서로 상충** → 양쪽 주장을 병기하고 직접 확인한 결과로 판정
   - 심각도(critical/major/minor)순 정렬, 각 항목에 확인됨/추정/미확인 표시
4. **저장 + 보고.** 종합 보고서를 `.harness/reviews/<시각>-<모드>-summary.md` 로 저장하고, 사용자에게는 critical/major 중심으로 요약해 보여준다. 발견이 없으면 "무엇을 검사했고 왜 통과인지"를 보고한다 — 그냥 "문제없음"이라고 하지 않는다.

## 규칙

- 리뷰어들의 주장을 그대로 믿지 않는다. critical/major 는 반드시 직접 파일을 열어 재확인한 뒤 보고한다 (하네스 표준: 사실/추정/미확인 구분).
- 두 모델 모두 구독 요금제로 호출되므로 추가 과금은 없지만, 사용량 한도를 소모한다. 같은 프로젝트에 하루 수회 이상 반복 실행하지 않는다.
- 한쪽 CLI가 실패(미설치/로그아웃/한도)하면 나머지 한쪽 결과로 진행하되, 교차 검증이 아니었음을 보고서에 명시한다.
