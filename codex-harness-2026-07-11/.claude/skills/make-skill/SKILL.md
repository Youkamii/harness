# make-skill — 설명을 스킬로 변환

목표: 사용자의 설명 → 하네스 저장소에 새 스킬 생성 → 검증 → 전역 설치 → 커밋·푸시까지 한 번에.

## 절차

1. **요구 파악.** 설명이 모호할 때만 최대 2~3개 질문한다: (a) 언제 쓰는 기능인가(트리거 상황), (b) 입력은 뭐고 결과물은 뭔가, (c) 매번 똑같이 반복되는 단계는 뭔가. 이미 명확하면 질문 없이 진행.
2. **하네스 위치 확인.** `harness dir` 명령의 출력(실패 시 `~/Git/Harness`)이 스킬의 원본 저장소다. 스킬은 반드시 거기에 만든다 — 현재 작업 폴더가 아니라.
3. **형태 결정.**
   - **지식형**: 절차·규칙·말투만 필요하면 → `SKILL.md` 하나로 충분
   - **도구형**: 결정적 로직(파싱, 변환, 파일 처리, API 호출)이 필요하면 → 스킬 폴더에 스크립트(`.mjs`/`.sh`)를 같이 두고, SKILL.md에는 "언제 어떤 인자로 그 스크립트를 실행하는지"를 쓴다
4. **작성.** `<하네스>/.claude/skills/<영문-kebab-이름>/SKILL.md`:
   - frontmatter `name`: 영문 kebab-case / `description`: 한국어로 **"무엇을 한다 + 언제 사용한다"** — 모델은 이 한 줄만 보고 호출 여부를 정하므로 사용자가 실제로 말할 법한 트리거 문구를 포함시킨다
   - 본문: 단계별 절차, 규칙, 출력 형식. **검증 단계를 반드시 포함**시킨다 (하네스 표준 규칙: 검증 없는 완료 선언 금지)
5. **검증.** 도구형이면 스크립트를 실제 실행해 테스트하고, 가능하면 스킬 절차 전체를 한 번 리허설한다. 검증 못 한 부분은 사용자에게 "미검증"이라고 명시한다.
6. **설치 + 배포.** `<하네스>/install.sh` 실행(전역 반영) → 하네스 저장소에 서술적 메시지로 커밋·푸시. 다른 PC는 `harness update`로 받는다.
7. **보고.** 스킬 이름, 호출 방법(`/이름` 또는 자연어 트리거), 그리고 **새 세션부터 자동 인식된다**는 점을 안내한다.

## 규칙

- **스킬 하나 = 한 가지 일.** 설명에 두 가지 일이 섞여 있으면 두 개로 쪼개자고 제안한다.
- 만들기 전에 `<하네스>/.claude/skills/` 목록을 확인한다 — 기존 스킬과 겹치면 새로 만들지 말고 기존 것을 수정한다.
- description에 "언제 사용"이 빠지면 그 스킬은 영원히 호출되지 않는다. 빼먹지 않는다.
- 검증 없이 "스킬 등록 완료"라고 말하지 않는다.
