# 하네스 아키텍처

> 이 저장소가 무엇이고, 어떤 파일이 무슨 역할을 하는지 설명하는 문서.
> 기술 배경 없이 읽을 수 있게 작성. 상세한 근거는 [harness-engineering-research.md](harness-engineering-research.md) 참고.

## 큰 그림

이 하네스는 밑바닥부터 만든 프로그램이 아니라, **월 구독에 포함된 Claude Code를 엔진으로 쓰는 개조 키트 + 오케스트레이터**다. API 추가 과금이 없다.

```
┌─────────────────────────────────────────────┐
│  계층 2: 오케스트레이터 (orchestrator/)        │  ← 장기 실행·자동화
│  claude -p 를 부품처럼 호출하는 우리 프로그램    │
├─────────────────────────────────────────────┤
│  계층 1: Claude Code 개조 (.claude/, CLAUDE.md)│  ← 행동 통제·프롬프트
│  hooks(집행), permissions(승인), skills(반복작업)│
├─────────────────────────────────────────────┤
│  엔진: Claude Code (Anthropic 제공, 구독 포함)  │  ← 우리가 안 만듦
└─────────────────────────────────────────────┘
```

- **대화형으로 쓸 때**: 평소처럼 `claude`를 실행하면 계층 1의 개조가 자동 적용된 상태로 동작한다.
- **자동/장기 실행할 때**: 계층 2의 오케스트레이터가 `claude -p`(헤드리스 모드)를 반복 호출하며 진행 상황을 파일로 관리한다.

## 디렉토리 구조

```
Harness/
├── CLAUDE.md                  # 에이전트에게 항상 주입되는 규칙·지침 (프롬프트 사전 설정)
├── .claude/
│   ├── settings.json          # 권한 규칙(무엇을 자동 허용/승인/차단), hooks 등록
│   ├── hooks/                 # 집행 스크립트 — 도구 호출을 가로채 검사·차단 (코드 레벨 강제)
│   │                          #   guard.js: 파괴적 명령 차단 / secrets-guard.js: 비밀 커밋 차단
│   │                          #   각각 *.test.mjs 회귀 테스트 동반 (harness test 로 실행)
│   ├── skills/                # 반복 워크플로 패키지 — /명령어 하나로 호출
│   └── agents/                # 커스텀 서브에이전트 정의 (예: 평가자 에이전트)
├── orchestrator/              # 장기 실행 러너 (Node 스크립트, 빌드 불필요)
│   ├── run.mjs                # 세션 루프: initializer → coder 반복, 한 세션에 한 기능
│   └── limits.mjs             # 구독 사용량 한도 감지 → 리셋 시각까지 대기 → 재개
├── evals/                     # 하네스 수정이 개선인지 측정
│   ├── run.mjs                # 러너: 태스크를 빈 임시 폴더에서 실행하고 check.sh로 채점
│   └── tasks/<이름>/          # 태스크 = task.md(지시) + check.sh(결정적 채점)
├── bin/harness                # 전역 명령 원본 (install.sh 가 ~/.local/bin/harness 로 설치)
├── scripts/                   # 설치 보조 (전역 settings.json 병합 등)
├── docs/
│   ├── architecture.md        # 이 문서
│   └── harness-engineering-research.md  # 기반 리서치
└── install.sh                 # PC당 1회 실행: 전역(~/.claude) 설치 + harness 명령 등록
```

## "X를 바꾸고 싶으면 어디를 보나"

| 하고 싶은 것 | 위치 | 형태 |
|---|---|---|
| 에이전트 말투·규칙·금지사항 바꾸기 | `CLAUDE.md` | 한국어 문장으로 작성 |
| 특정 명령을 승인받게 / 차단하기 | `.claude/settings.json` | 규칙 한 줄 추가 |
| 프롬프트로 안 되는 강제 규칙 (예: 커밋 전 테스트 필수) | `.claude/hooks/` | 작은 스크립트 |
| 자주 시키는 작업을 명령어로 만들기 | `.claude/skills/` | 마크다운 파일 |
| 작업자/평가자 분리 등 역할 에이전트 | `.claude/agents/` | 마크다운 파일 |
| 밤새 돌릴 작업의 동작 방식 | `orchestrator/` | Node 스크립트 |
| 수정 전후 성능 비교 | `evals/` | 테스트 작업 추가 |

직접 고쳐도 되고, Claude Code에게 "○○ 바꿔줘"라고 시켜도 된다. 어느 쪽이든 다음 실행부터 반영된다.

## 적용 범위

`install.sh`를 PC당 1회 실행하면 전역(`~/.claude/`)에 설치되어 **모든 폴더의 `claude`에 자동 적용**되고,
`harness` 명령(`~/.local/bin/harness`)이 생긴다. 이 저장소가 원본(source of truth)이며,
하네스를 수정하면 `harness update`(git pull + 재설치)로 어느 PC에서든 반영한다.

## 설계 원칙 (리서치에서 검증된 것)

1. **부탁이 아닌 집행** — 중요한 규칙은 CLAUDE.md(부탁)가 아니라 hooks/permissions(집행)로.
2. **세션 간 기억은 파일로** — 장기 작업은 진행 로그·기능 목록(JSON)·git 히스토리로 이어간다. (컨텍스트 압축보다 리셋+아티팩트가 우수)
3. **작업자와 평가자 분리** — 에이전트의 자기 평가를 믿지 않는다. 별도 평가자가 검증.
4. **한 번에 한 기능** — 원샷 시도, 조기 완료 선언, 미검증 완료 표시가 3대 안티패턴.
5. **모든 개조에 이유를 기록** — 각 hook/규칙이 "모델의 어떤 한계 때문에 존재하는지" 주석으로 남긴다. 모델이 좋아지면 제거 후보가 된다.
6. **수정은 측정으로 검증** — 하네스를 고치면 evals로 전후 비교.

## 비용 구조

- 모든 모델 호출은 로그인된 구독 계정(Pro/Max)으로 처리 — **API 추가 과금 없음.**
- 구독에는 사용량 한도(5시간 창 + 주간)가 있다. 대화형 사용은 거의 영향 없음.
- 장기 실행은 한도를 소모하므로 `orchestrator/limits.ts`가 한도 도달 시 대기 후 재개한다. 추가 요금은 발생하지 않고, 최악의 경우 "잠시 멈춤"이다.
