# kickoff — 기능 개발 파이프라인

계획 없이 코드부터 짜는 것을 구조적으로 금지한다. 아래 단계는 순서대로, 건너뛰지 않는다.
전 과정에서 **사용자에게 묻지 않는다** — 되돌릴 수 있는 결정은 스스로 내리고 "결정사항"으로 기록한다.
(예외: 비용 발생·되돌리기 어려운 파괴·외부 공개는 반드시 확인)

## 0. 요구사항 해석 (스스로)

- 요구를 3줄 이내로 요약하고, 기능별 **완료 기준(DoD)** 을 정한다: "X를 실행하면 Y가 관측된다" 형태로.
- 모호한 부분은 가장 합리적인 해석을 골라 진행하고 해석을 명시한다. 질문으로 사용자를 막지 않는다.

## 1. 태스크 분해

- TaskCreate로 **기능 단위** 태스크를 등록한다 (기능 1개 = 태스크 1개, 3~7개 권장).
- 의존관계가 있으면 TaskUpdate의 addBlockedBy로 연결한다.
- 작업 시작 시 in_progress, 검증 통과 시에만 completed. 실패/부분 구현 상태로 completed 금지.

## 2. GitHub 이슈 생성

- 대상이 git 저장소이고 GitHub 원격이 있으며 `gh auth status`가 살아 있으면, 기능별로 이슈를 만든다:
  ```bash
  gh issue create --title "[기능] <이름>" --body "## DoD
- <완료 기준>
## 계획
- <단계>"
  ```
- 이슈 번호는 **커밋 메시지(#N)와 최종 보고에** 기록한다. 세션 태스크는 세션이 끝나면 사라지므로 기능↔이슈 매핑의 영속 기록은 커밋·이슈 본문이 담당한다.
- gh를 못 쓰는 환경이면 이 단계는 생략하고 태스크로만 추적한다 (생략했음을 보고에 명시).
- 본인 소유 저장소의 이슈 생성/닫기·푸시는 일상 작업이다 — 사용자 확인 없이 진행한다 (타인·조직 저장소는 헌법의 '외부 공개' 예외).

## 3. 구현 — 한 번에 한 기능

- 가장 작은 다음 기능 하나만 완성한다. 여러 기능을 한 번에 끝내려 하지 않는다.
- **기능별 커밋**: 기능 하나가 검증되면 즉시 커밋. 메시지 제목에 이슈 연결(`#N`), 본문은 다음 세션이 커밋만 읽고 이해할 수 있게 무엇을/왜.
- 비밀(API 키·암호)은 .env로 분리하고 .gitignore 확인 (secrets-guard 훅이 이중 차단하지만 훅은 면죄부가 아니다).

## 4. 자가검증 — 실행한 것만 완료다

- 코드를 읽고 "될 것 같다"로 판정하지 않는다. 실제로 실행(테스트/앱 구동/스크립트)해서 DoD가 관측되는지 확인한다.
- 검증 환경이 복잡하면 verifier 서브에이전트(Agent tool, subagent_type: verifier)에게 맡긴다.
- 실행할 수 없는 항목은 **미검증**으로 명시한다. 미검증을 완료로 보고하지 않는다.

## 5. 멀티에이전트 적대 리뷰

- 모든 기능 구현이 끝나면(또는 큰 기능 하나가 끝날 때마다) **/red-review 스킬의 절차**로 적대 리뷰를 돈다:
  red-reviewer 서브에이전트를 서로 다른 렌즈(정합성·보안·단순화)로 **병렬 3개** 소환 → 발견을 직접 재확인 → CONFIRMED만 수정.
- 리뷰어 주장을 그대로 믿지 않는다. critical/major는 반드시 해당 파일을 직접 열어 재확인한다.

## 6. 마무리

- CONFIRMED 지적 수정 → 수정 커밋 → **푸시** → 이슈 닫기:
  ```bash
  git push && gh issue close <N> --comment "완료: <검증 방법 요약>"
  ```
- 최종 보고 형식: **결정사항**(스스로 내린 판단과 근거) / **확인됨**(실행으로 검증) / **미검증**(사유) / **리뷰 결과**(발견 n건 중 수정 m건, 기각 사유).
